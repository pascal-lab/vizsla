Vide provides these reading features:

- `Go to Definition` / `Go to Declaration`: jump to symbols, modules, ports, parameters, or instantiated targets.
- `Find References`: find symbol references.
- Hover: inspect signatures for modules, ports, parameters, tasks, and functions.
- Document Highlight: highlight uses of the same symbol in the current file.

Related settings reference: [Scope](../../../advanced-guide/vscode-settings/#scope), [Semantic Tokens](../../../advanced-guide/vscode-settings/#semantic-tokens).

## Definition Does Not Jump

Run `Go to Definition` on the instance module name first. If the module name itself does not jump, the target file is usually not loaded through project configuration:

- Does `sources` match the file that defines the target module?
- Does `include_dirs` include the include directory?
- Do macro definitions, configured through `defines`, enable the conditional-compilation branch that contains the target? If the target module is hidden behind an inactive `` `ifdef ``, it is not analyzed in the current project view.
- Does `libraries` include the dependency project?

See [Project Configuration](../../project-configuration/) for the fields.

## Duplicate Modules Jump Somewhere Unexpected

Best-effort indexing can make a nearest-candidate guess for reading, but this is not a SystemVerilog language rule. In real projects, handle duplicate module names through explicit project configuration, library boundaries, or your build scripts. For the exact behavior without full configuration, see [Navigation and Duplicate Modules](../../../advanced-guide/parsing-and-analysis/#navigation-and-duplicate-modules).

## References or Rename Miss Files

Confirm that the current file belongs to the same project view. With only one file open, Vide may only handle that file or locally indexed information. Add the related sources to `sources`, then run `Vide: Reload Project Configuration`.
