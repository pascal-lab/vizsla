The VS Code extension registers two language modes:

| Language | File extensions |
| --- | --- |
| Verilog | `.v`, `.vh` |
| SystemVerilog | `.sv`, `.svh`, `.svi` |

After you open these files, VS Code applies basic syntax highlighting, comment rules, bracket matching, and file icons. These do not depend on the background language server, so basic editing remains available while the server is starting.

Diagnostics, go to definition, completion, hover, signature help, inlay hints, semantic highlighting, outline, folding, selection ranges, and instance count lens entries become available after the language server starts. Cross-file understanding also needs enough project configuration at the workspace root.

Related settings reference: [Semantic Tokens](../../../advanced-guide/vscode-settings/#semantic-tokens), [Files](../../../advanced-guide/vscode-settings/#files).

## What Semantic Highlighting Adds

Syntax highlighting only knows lexical information such as keywords, numbers, and comments. Semantic highlighting uses language-server analysis. When your theme supports semantic tokens, Vide can additionally mark port direction, clock/reset ports, and symbol read/write positions.

The useful cues are small but practical:

- `input` and `output` ports can use different colors for signal direction.
- clk/rst ports can stand out, making timing boundaries easier to scan in module interfaces.
- Read and write positions can be styled differently, which helps when checking where a signal is assigned or consumed.

## Header Files

`.vh`, `.svh`, and `.svi` files usually participate through the `.v` or `.sv` file that includes them. When you open a header by itself, Vide still provides syntax highlighting and local editing features; put the including source files in `sources` and the header directory in `include_dirs` so cross-file analysis uses the real project view.
