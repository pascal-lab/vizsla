Diagnostics appear in the VS Code `Problems` panel and as editor underlines. Syntax and parse errors usually only need the current file; cross-file semantic diagnostics need project information from `vide.toml`, such as `sources`, `include_dirs`, `defines`, and `libraries`.

Diagnostic refresh and diagnostic rules are VS Code Settings. Search for `Vide Diagnostics` in the Settings UI, or write `vide.diagnostics.*` keys in user or workspace `settings.json`; see [Diagnostics](../../../advanced-guide/vscode-settings/#diagnostics) for the full reference.

## Refresh Policy

The default `vide.diagnostics.update` value is `onSave`, so diagnostics refresh when you save. Keep this default for large projects to avoid heavy analysis on every edit.

For diagnostics while typing, add this to VS Code user or workspace `settings.json`:

```json
{
  "vide.diagnostics.update": "onType"
}
```

:::note[Project configuration changes]
After changing `vide.toml`, run `Vide: Reload Project Configuration` to reread project configuration. Headers opened by themselves, or files that are not matched by `sources`, usually do not run cross-file diagnostics on their own; see [Project Configuration](../../project-configuration/).
:::

## Adjust Diagnostic Rules

Place the cursor on the diagnostic and open the lightbulb menu. For slang diagnostics with an identifiable code, Vide provides quick fixes that write a rule to user or workspace settings, such as ignoring that diagnostic type or downgrading an error to a warning.

You can also edit `vide.diagnostics.slang.rules` manually in VS Code Settings. The rule format is documented in the [Diagnostics settings reference](../../../advanced-guide/vscode-settings/#diagnostics).
