Vide can provide syntax highlighting and basic editing after opening any Verilog/SystemVerilog project. To make cross-file navigation, completion, diagnostics, macro-branch selection, and Qihe argument derivation more accurate, add a project manifest named `vide.toml` to the workspace root.

This page only covers the first-time setup flow. For complete field details and edge cases, see [Project Configuration Reference](../project-configuration/).

## 1. Create the Project Manifest

Open your RTL project directory in VS Code. If the directory contains RTL files but no `vide.toml`, Vide prompts you to create a project manifest. You can also click the `Vide` status item in the lower-right status bar and choose `Create Manifest`.

:::note
Later pages use “workspace root” to mean this top-level directory opened in VS Code.
:::

The default template creates an empty project configuration:

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = []

# include_dirs = ["include"]
# defines = ["SYNTHESIS"]
# top_modules = ["top"]
# libraries = ["../common_cells"]
# exclude = ["build/**"]
```

The next sections explain what each field does.

The default `sources = []` is an unfinished state: it tells Vide not to scan the whole workspace yet. Replace it with real paths such as `rtl/**`, and Vide will analyze the project according to that structure.

## 2. Describe the Project Layout

Assume your project looks like this:

<FileTree>
- my-rtl/
  - vide.toml
  - rtl/
    - top.sv
    - counter.sv
  - include/
    - defs.svh
  - sim/
    - work/
  - build/
</FileTree>

A first manifest can be:

```toml
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS"]
top_modules = ["top"]
exclude = ["build/**", "sim/work/**"]
```

These fields tell Vide:

- `sources`: which RTL source files belong to the project. Small projects can start with the source directory, such as `rtl/**`.
- `include_dirs`: where to search for files used by `` `include "defs.svh" ``.
- `defines`: which macro branches are predefined, such as `"SYNTHESIS"` or `"DATA_WIDTH=32"`.
- `top_modules`: the top-level module names for this project, which tell Vide where to start analysis.
- `exclude`: generated directories, simulation output, and cache directories to ignore.

:::note
Always write paths with `/`, even on Windows. Use `rtl/**`, not `rtl\**`.
:::

## 3. Reload the Project

After you save `vide.toml`, Vide reloads project configuration automatically by default. In practice there are three states:

- No `vide.toml`: Vide scans the workspace for reading features, but does not create a project-level compilation profile.
- The empty default template: `sources = []` tells Vide not to scan the whole workspace yet, because the real source paths still need to be filled in.
- A configured `vide.toml`: Vide understands the project through `sources`, `include_dirs`, `defines`, `libraries`, and related fields.

## What Changes After Configuration

| Area | Without a project manifest | After `vide.toml` is configured |
| --- | --- | --- |
| [Go to definition](../daily-use/navigation/), [find references](../daily-use/references/), and [hover](../daily-use/hover/) | Scans the workspace and provides the jumps, references, and symbol information needed while reading code | Resolves targets through `sources`, `libraries`, include directories, and macro branches |
| [Completion](../daily-use/completion/) and [signature help](../daily-use/signature-help/) | Uses the current file and scanned files for candidates; shows port/parameter help once the target module is resolved | Uses target modules, include directories, and macro branches for candidates and current-position hints |
| [Rename](../daily-use/rename/) and [quick fixes](../daily-use/quick-fixes/) | Local code actions are available; project rename is disabled | Fills or converts ports and parameters, and updates declarations, references, and connections across the project |
| [Syntax highlighting](../daily-use/syntax-highlighting/) and [semantic highlighting](../daily-use/semantic-highlighting/) | Syntax highlighting is available, and the current file gets base semantic tokens | Uses project analysis to mark port directions, read/write positions, and macro branches |
| [Inlay hints](../daily-use/inlay-hints/), [document symbols](../daily-use/document-symbols/), [folding](../daily-use/folding/), and [selection range](../daily-use/selection-range/) | The current file gets outline, folding, and selection ranges; inlay hints are shown after the target module is resolved | Shows port, parameter, outline, folding, and selection-range information from the configured project |
| [Diagnostics](../daily-use/diagnostics/) | Open files get syntax and parse diagnostics; project-level semantic diagnostics are not created | Uses `sources`, `include_dirs`, and `defines` for cross-file semantic diagnostics |
| [Qihe](../daily-use/qihe/) | Uses the current file as the analysis input | Derives compile arguments from `sources`, `top_modules`, `include_dirs`, and `defines` |
