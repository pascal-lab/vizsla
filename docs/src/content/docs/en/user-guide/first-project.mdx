Vide can provide syntax highlighting and basic editing after opening any Verilog/SystemVerilog project. To make cross-file navigation, completion, diagnostics, macro-branch selection, and Qihe argument derivation more accurate, add a project configuration file named `vide.toml` to the workspace root.

This page walks through a lightweight first configuration. For complete field details and boundary rules, see [Project Configuration Reference](../project-configuration/).

## 1. Create the Project Configuration File

Open your RTL project directory in VS Code. If the directory contains RTL files but no `vide.toml`, Vide prompts you to create a project configuration file. You can also click the `Vide` status item in the lower-right status bar and choose `Create Project Configuration File`.

:::note
Later pages use “workspace root” to mean this top-level directory opened in VS Code.
:::

The default template creates an empty project configuration:

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = []

# include_dirs = ["include"]
# defines = ["SYNTHESIS"]
# top_modules = ["top"]
# libraries = ["../common_cells"]
# exclude = ["build/**"]
```

:::note
- The default `sources = []` is an unfinished state: it tells Vide not to scan the whole workspace yet. Replace it with real paths such as `rtl/**`, and Vide will analyze the project according to that structure.
- The leading `#:schema ...` line is a TOML comment. It can help editors identify the configuration format.
:::

## 2. Describe the Project Layout

Assume your project looks like this:

<FileTree>
- my-rtl/
  - vide.toml
  - rtl/
    - top.sv
    - counter.sv
  - include/
    - defs.svh
  - sim/
    - work/
  - build/
</FileTree>

A first configuration can be:

```toml
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS"]
top_modules = ["top"]
exclude = ["build/**", "sim/work/**"]
```

These fields tell Vide:

- `sources`: **required**, selects which RTL source files belong to this project. For example, `rtl/**` recursively matches every file under the directory, while `rtl/*.sv` only matches `.sv` files directly under `rtl`.
- `include_dirs`: **recommended**, tells Vide where to search for headers used by `` `include "defs.svh" ``. For example, `include`.
- `defines`: **optional**, predefines macro branches, such as `"SYNTHESIS"` or `"DATA_WIDTH=32"`.
- `top_modules`: **recommended**, declares the top-level module names for this project and tells Vide where to start analysis.
- `exclude`: **optional**, filters out generated directories, simulation output, and cache directories so Vide does not analyze them. In this example, `exclude` can also be omitted because `sources` is not affected by those paths.

:::note
Always write paths with `/`, even on Windows. Use `rtl/**`, not `rtl\**`. All paths must stay inside the workspace root.
:::

## 3. A More Realistic Project Example

The `my-rtl` project above is intentionally small. Now use a real project, [darklife/darkriscv](https://github.com/darklife/darkriscv), as an example. The repository root contains `darkriscv.core`, which defines project information. A corresponding `vide.toml` can be:

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json

top_modules = ["darksocv"]
sources = [
  "rtl/*.v",
]
include_dirs = [
  "rtl",
  "src",
]
```

The mapping is:

- `top_modules = ["darksocv"]`: comes from `toplevel: darksocv` in the `.core` default target.
- `sources = ["rtl/*.v"]`: covers module definitions used directly or indirectly by `darksocv`, including `darkpll.v`, `darkbridge.v`, `darkram.v`, `darkio.v`, `darkriscv.v`, and `darkuart.v`.
- `include_dirs = ["rtl", "src"]`
  - `rtl/config.vh` is an include file, so `rtl` is an include directory.
  - `src/darksocv.mem` is simulation or initialization data used by the source code, so put `src` in `include_dirs`.

The configuration above only covers the synthesizable part. If you also want to analyze simulation code, how should `vide.toml` be written?

For example, to analyze the simulation entry from the `.core` `sim` target, add `sim/darksimv.v` from the `tb` fileset and change the top module to `darksimv`. The `sim` target's icarus configuration uses `-g2012`, and `config.vh` switches simulation branches based on `__ICARUS__`, so define that macro as well:

```toml
sources = [
  "rtl/*.v",
  "sim/darksimv.v",
]

include_dirs = [
  "rtl",
  "src",
  "sim",
]

defines = ["__ICARUS__"]
top_modules = ["darksimv"]
```

Board targets are configured the same way. For example, the `colorlight_i5` target uses the `rtl` and `colorlighti5` filesets, and enables the `__YOSYS__` and `LATTICE_ECP5_COLORLIGHTI5` vlog defines. The corresponding configuration can be:

```toml
sources = [
  "rtl/*.v",
  "boards/colorlighti5/pll_ref_25MHz.v",
]

include_dirs = ["rtl", "src"]
defines = ["__YOSYS__", "LATTICE_ECP5_COLORLIGHTI5"]
top_modules = ["darksocv"]
```

## 4. Project Status

After you save `vide.toml`, Vide reloads project configuration automatically by default.

In practice there are three project states:

- No `vide.toml`: Vide scans the workspace for reading features, but does not create a project-level compilation profile.
- The empty default template: `sources = []` tells Vide not to scan the whole workspace yet, because the real source paths still need to be filled in.
- A configured `vide.toml`: Vide understands the project through `sources`, `include_dirs`, `defines`, `libraries`, and related fields.

For details, each [feature page](../features/) explains the behavior with and without project configuration. We also summarize the differences in [What Changes After Configuration](../project-configuration/#what-changes-after-configuration).
