This page is the `vide.toml` project configuration reference. For first-time setup, start with [Configure the First Project](../first-project/) to write a working project configuration file.

## Field Quick Reference

A complete small-project `vide.toml` usually looks like this:

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS", "FEATURE_PIPELINE=1"]
top_modules = ["top"]
libraries = ["vendor/common"]
exclude = ["build/**", "sim/work/**"]
```

If the project has no external libraries or generated directories, remove `libraries` and `exclude`; the other fields usually affect navigation, completion, diagnostics, rename, and Qihe analysis directly.

| Field | Type | Purpose |
| --- | --- | --- |
| `sources` | Array of globs, or omitted | Selects project source files. A common value is `["rtl/**"]`. |
| `include_dirs` | Array of paths, or omitted | Include search directories, for example `["include"]`. |
| `defines` | Array of strings | Predefined macros. Supports both `"NAME"` and `"NAME=value"`. |
| `top_modules` | Array of strings | Declares top-level modules for the current project. |
| `libraries` | Array of paths | External libraries or dependency project directories. Vide loads them separately and resolves them while analyzing the current project. |
| `exclude` | Array of globs | Filters generated directories, simulation output, or black-box files out of loaded files. |

## Path and Glob Rules for `sources` and `exclude`

`sources` and `exclude` are not plain directory paths. They are path patterns with wildcards, also called globs. The two most common wildcards are:

- `*`: matches names within one directory. For example, `rtl/*.sv` only matches `.sv` files directly under `rtl`.
- `**`: matches across nested directories. For example, `rtl/**` continues matching files inside subdirectories under `rtl`.

:::note
For background, see this [glob overview](https://en.wikipedia.org/wiki/Glob_(programming)).
:::

Vide applies these rules to `sources` and `exclude`:

- Patterns are relative to the workspace root, for example `rtl/**`.
- Use `/` as the path separator. Do not write `rtl\**`.
- Absolute paths and any pattern containing a `..` path component are rejected.
- `*` does not cross directories; use `**` when you need recursive matching.

In `include_dirs` and `libraries`, values are ordinary paths resolved relative to the directory that contains the current project configuration file. `include` and `include/` refer to the same directory; for consistency, this documentation uses `include`.

## Field Details

### sources

`sources` has three common states:

| Form | Result |
| --- | --- |
| Omitted | Vide creates a best-effort workspace index for code-reading features such as navigation, references, and hover, but does not treat it as a fully configured project. |
| `sources = []` | Disables automatic workspace scanning. This is useful when you want to open the project safely and configure it later. |
| `sources = ["rtl/**"]` | Loads matching files with globs and adds them to project analysis. |

A short way to remember it:

- Omitted `sources`: scan the workspace for code reading, but do not treat the scan result as the configured project.
- `sources = []`: do not scan source files automatically.
- `sources = ["rtl/**"]`: these files are the current project sources.

Common examples:

```toml
sources = ["rtl/**", "ip/**/*.sv", "tb/**/*.sv"]
```

The current project loader reads these extensions:

- `.v`
- `.sv`
- `.vh`
- `.svh`
- `.svi`
- `.map`

### include_dirs

`include_dirs` tells Vide which directories to search for `file.svh` in `` `include "file.svh" ``.

If `sources` is set but `include_dirs` is not configured explicitly, Vide infers a default include directory from `sources`. For example, `sources = ["rtl/**/*.sv"]` uses `rtl` as the default include directory. If `sources` is omitted, Vide does not infer include directories from best-effort indexing.

In all cases, an explicit `include_dirs = []` means forcing the include search path to be empty; Vide will not infer a default.

`include_dirs` are include search paths only. They do not mean every file in those directories becomes an independent compile entry. Vide analyzes `.v`/`.sv` code after included headers are expanded. Opening a header directly, or listing its directory in `include_dirs`, does not mean Vide runs full standalone project diagnostics for that header. As a result, features that depend on full project context are usually less complete inside header files than inside `.v`/`.sv` source files.

### defines

Each entry in `defines` is one macro definition:

```toml
defines = [
  "SYNTHESIS",
  "WIDTH=32",
  "MODE=fast sim",
]
```

Macro names must be valid identifiers. Macros with values are parsed as `NAME=value`; `value` may contain spaces.

### libraries

`libraries` declares dependency libraries. Paths are resolved relative to the directory that contains the current project configuration file. Each library path is loaded as another workspace:

- If the library directory contains `vide.toml`, Vide loads the library with that project configuration file.
- If the library directory has no project configuration file, Vide loads it as an unconfigured library.
- Dependencies participate in the current project's analysis, and transitive dependencies are supported.

`libraries` does not replace the current project's own `sources`. The current workspace's Verilog/SystemVerilog source files should still be declared through `sources`; `libraries` only describes external projects or library directories referenced by the current project.

### exclude

`exclude` uses the same workspace-relative glob syntax described above. It is useful for generated directories, simulation output, cache directories, or black-box stubs:

```toml
exclude = ["build/**", "out/**", "sim/work/**", "**/*_bb.v"]
```

`exclude = ["build"]` only matches the path named `build` itself and does not recursively exclude the directory contents. Use `build/**` for recursive directory exclusion.

`exclude` only controls which files Vide selects for project analysis. If you use VS Code and also want VS Code itself to watch fewer large directories, configure VS Code's `files.watcherExclude` separately. These settings do not replace each other: the former controls what Vide analyzes, while the latter controls what the VS Code file watcher pays less attention to.

## Project File Discovery Rules

- The project configuration file name must be `vide.toml`.
- Vide does not automatically discover project configuration files from subdirectories. When you open `repo`, it only checks `repo/vide.toml`.

Feature behavior before and after project configuration is summarized in [Behavior With and Without Project Configuration](../project-configuration-effects/).
