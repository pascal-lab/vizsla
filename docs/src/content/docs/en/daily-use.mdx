This page introduces Vizsla by daily IDE workflow. It is not the full settings reference; use [VS Code Settings](./vscode-settings.md) for every setting and [Project Configuration](./project-configuration.md) for project semantics.

## Language Detection

The VS Code extension registers two languages:

| Language | File extensions |
| --- | --- |
| Verilog | `.v`, `.vh` |
| SystemVerilog | `.sv`, `.svh`, `.svi` |

After opening one of these files, VS Code first applies Vizsla's TextMate grammar and language configuration. Basic syntax highlighting, comment rules, and bracket matching do not depend on the language server starting successfully.

After the language server starts, diagnostics, navigation, completion, hover, signature help, inlay hints, semantic tokens, outline, folding, selection ranges, and code lens are connected. Cross-file features need enough project information from the current workspace.

## Diagnostics

During daily editing, diagnostics appear in the VS Code `Problems` panel and as editor underlines. Vizsla provides both single-file parse diagnostics and semantic diagnostics that need project information.

Diagnostics are enabled by default, and `vizsla.diagnostics.update` defaults to `"onSave"`. Large RTL projects usually benefit from keeping this default. If you explicitly want refreshes while typing, change the setting to `"onType"`.

Quick fixes usually appear with the related diagnostic. If an expected fix is missing, first confirm that the diagnostic has refreshed, then check whether the instance target, includes, defines, and libraries are loaded by the current project.

For slang warnings, severity overrides, and refresh timing, see [Diagnostics in VS Code Settings](./vscode-settings.md#diagnostics). For how project files affect semantic diagnostics, see [Project Configuration](./project-configuration.md) and [Parsing and Analysis Model](./parsing-and-analysis.md).

## Navigation and Reading

For RTL reading, the common entry points are `Go to Definition`, `Go to Declaration`, `Find References`, document highlights, and rename:

- Jump from an instance name to the target module.
- Jump from a signal, port, parameter, typedef, function, or task to its declaration or definition.
- Find references to the same symbol in the current project.
- Highlight related locations in the current file when the cursor is on a symbol.
- When renaming, let `Prepare Rename` check whether the current position is renameable before reference search generates a workspace edit.

These features depend more on the semantic index than plain text search does. When results are unexpected, first check whether the target files are loaded by the project configuration and whether `vizsla.scope.visibility` matches your scope expectations.

## Editing Assistance

Completion appears at common Verilog/SystemVerilog input positions, such as module bodies, members after `.`, instance port lists, parameter lists, event controls, preprocessor directives, and system task/function positions.

Vizsla chooses completion sources by the current position instead of dumping every candidate into the list. Common results include preprocessor directives, keywords and snippets, expression candidates, members after `.`, instance ports and parameters, signals in event controls, and system tasks/functions. It avoids unrelated candidates inside comments, strings, and literals where possible. Port and parameter completion require the instantiated target module to be resolvable.

Hover first identifies names and literals:

- For symbol names, Vizsla resolves the definition and renders information about modules, ports, parameters, declarations, instances, functions, and other items.
- For port connection shorthand, Vizsla shows information for both the port side and the local side.
- For literals, Vizsla renders parsed literal information.

Signature help triggers on `(`, `,`, and `.`. It mainly serves module instance port connections and parameter assignment lists: while you type a connection, you can see the target module's port or parameter order, and the active parameter follows the cursor. Use `vizsla.signature.help.params.only` when you only want parameter-related signatures.

Inlay hints are enabled by default to reduce mechanical trips back to the port list:

- Ordered port connections or empty connections show target port names.
- Ordered parameter assignments show target parameter names.
- Structure endings show the structure name.

Code actions appear next to the problems or edits they can solve:

- If the `Problems` panel reports missing instance connections, such as unconnected named or unnamed ports, place the cursor near the instance connection list and use `Fill connections`.
- If a parameter has no value, use `Fill parameters`.
- If ports or parameters mix ordered and named style, use `Convert ordered port connections to named connections`, `Convert ordered parameter assignments to named assignments`, or `Remove empty port connections`.
- If `.port` is missing explicit `()`, use `Add explicit empty port connection`. If an instance has no port list at all, use `Add empty instance port list`.
- When the cursor is on an integer literal, Vizsla may offer `Convert literal to binary`, `Convert literal to decimal`, `Convert literal to hexadecimal`, and related base conversions.

These actions need Vizsla to understand the current position. Instance port and parameter actions also need the target module to be resolvable. If the target module itself cannot be found, fix `sources`, `include_dirs`, `defines`, or `libraries` first. Vizsla currently does not provide a dedicated automatic fix for unresolved modules or imports.

Formatting entry points include `Format Document`, `Format Selection`, and on-type formatting when pressing Enter. The only current formatter provider is `verible`, which calls the external `verible-verilog-format` executable. If it is not on `PATH`, configure an absolute path with `vizsla.formatter.path`. Formatter arguments and Enter behavior settings are covered in [Formatter and Formatting in VS Code Settings](./vscode-settings.md#formatter-and-formatting).

## Visual and Structure Assistance

Semantic tokens add semantic roles on top of TextMate highlighting. The default settings enable clock/reset port modifiers, input/output port modifiers, and Vizsla-contributed default styling for the `read` and `write` modifiers. When your theme supports the corresponding tokens, port direction, clock/reset ports, and ordinary symbols are easier to distinguish.

Outline and folding ranges are aimed at reading long files. Outline shows symbols such as modules, ports, parameters, declarations, instances, blocks, functions, and generate blocks. Folding collapses structures such as modules, blocks, statements, and comments.

Selection ranges support VS Code's expand/shrink selection commands, growing the current token into an expression, statement, block, or larger syntax structure.

Instance code lens is enabled by default and shows `0 instances`, `1 instance`, or `N instances` at module declarations. The current code lens only shows the instance count and does not bind a navigation command. Use `Find References` when you need to locate the actual instances.

## Common Requirements and Limits

The full cross-file experience depends on project configuration. The default `vizsla.toml` created by VS Code contains `sources = []`, so it does not scan the workspace. To enable more accurate semantic diagnostics, port/parameter completion, and instance-related assistance, add `sources`, `include_dirs`, `defines`, `libraries`, or `top_modules` for your project layout. See [Project Configuration](./project-configuration.md) for details.

When includes, macros, libraries, or target modules are missing, Vizsla keeps as many single-file features as it can, but cross-file diagnostics, instance signatures, port/parameter completion, inlay hints, code actions, and code lens may be reduced or unavailable.

For daily troubleshooting, start with `Vizsla: Show Status`, `Vizsla: Show Language Server Output`, `Vizsla: Reload Project Configuration`, `Vizsla: Restart Language Server`, and `Vizsla: Show Server Version` from the command palette. Qihe analysis and diagnostics profiling commands are documented in [Commands, Status, and Logs](./commands-status-logs.md).

If behavior differs from this page, check [Troubleshooting](./troubleshooting.md) first. It covers server startup, project loading, diagnostics refresh, formatter failures, and watcher refresh issues.
