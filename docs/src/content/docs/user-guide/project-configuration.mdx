这是一页 `vide.toml` 项目配置参考。第一次配置工程时，先按 [配置第一个项目](../first-project/) 写出能工作的项目配置文件。

## 字段速查

一个完整小工程的 `vide.toml` 通常长这样：

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS", "FEATURE_PIPELINE=1"]
top_modules = ["top"]
libraries = ["vendor/common"]
exclude = ["build/**", "sim/work/**"]
```

如果工程没有外部库或生成目录，可以删掉 `libraries` 和 `exclude`；其余字段一般会直接影响跳转、补全、诊断、重命名和骑河分析。

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `sources` | glob 数组或省略 | 选择工程源文件。常用写法是 `["rtl/**"]`。 |
| `include_dirs` | 路径数组或省略 | include 搜索目录，例如 `["include"]`。 |
| `defines` | 字符串数组 | 预定义宏。支持 `"NAME"` 和 `"NAME=value"`。 |
| `top_modules` | 字符串数组 | 声明当前工程的顶层模块名。 |
| `libraries` | 路径数组 | 外部库或依赖工程目录。Vide 会单独加载它们，并在分析当前工程时解析依赖。 |
| `exclude` | glob 数组 | 从已加载文件中过滤生成目录、仿真输出或黑盒文件。 |

## `sources` 和 `exclude` 的路径和 glob

`sources` 和 `exclude` 不是普通目录路径，而是“带通配符的路径写法”，也叫 glob。最常用的两个通配符是：

- `*`：匹配一层目录里的名字，例如 `rtl/*.sv` 只匹配 `rtl` 目录正下方的 `.sv` 文件。
- `**`：匹配多层目录，例如 `rtl/**` 会继续匹配 `rtl` 子目录里的文件。

:::note
想了解这个概念，可以看 [glob 简介](https://en.wikipedia.org/wiki/Glob_(programming))。
:::

Vide 对 `sources` 和 `exclude` 有几条固定规则：

- 必须相对工作区根目录，例如 `rtl/**`。
- 路径分隔符统一写 `/`，不要写 `rtl\**`。
- 不支持绝对路径，也不支持路径里出现 `..`。
- `*` 不跨目录；需要递归目录时用 `**`。

在 `include_dirs` 和 `libraries` 里值是普通路径，并且相对当前项目配置文件所在目录解析。`include` 和 `include/` 表示同一个目录；为了统一，文档里推荐写成 `include`。

## 字段详解

### sources

`sources` 有三种常见状态：

| 写法 | 结果 |
| --- | --- |
| 省略 | Vide 会尽力索引工作区，用于跳转、引用、悬停等阅读功能，但不会把它当成完整工程配置。 |
| `sources = []` | 不自动扫描工作区。用于先打开工程，之后再手动补配置。 |
| `sources = ["rtl/**"]` | 按 glob 加载匹配文件，并让这些文件进入项目分析。 |

可以记成：

- 省略 `sources`：先帮我读一下工作区，但不要把扫描结果当成正式工程。
- `sources = []`：不要自动扫描源码。
- `sources = ["rtl/**"]`：这些文件就是当前工程源码。

常用写法：

```toml
sources = ["rtl/**", "ip/**/*.sv", "tb/**/*.sv"]
```

当前工程加载器读取这些扩展名：

- `.v`
- `.sv`
- `.vh`
- `.svh`
- `.svi`
- `.map`

### include_dirs

`include_dirs` 用于告诉 Vide 去哪些目录查找 `` `include "file.svh" `` 中的 `file.svh`。

如果你已经写了 `sources`，但没有写 `include_dirs`，Vide 会从 `sources` 推导一个默认 include 目录。例如 `sources = ["rtl/**/*.sv"]` 会把 `rtl` 作为默认 include 目录。但如果 `sources` 被省略，Vide 不会从尽力索引结果反推 include 目录。

任何情况下，显式写成 `include_dirs = []` 时，都表示强制使用空 include 目录，Vide 不会进行推导。

`include_dirs` 只表示 include 搜索路径，不表示这些目录里的文件都是独立编译入口。Vide 实际分析的是引入头文件后的 `.v`/`.sv` 代码；只打开头文件，或者只把目录写进 `include_dirs`，不等于 Vide 会为这个头文件单独跑完整工程诊断。因此在头文件中，依赖完整工程上下文的功能通常不如在 `.v`/`.sv` 源文件中完整。

### defines

`defines` 中每一项都是一个宏定义：

```toml
defines = [
  "SYNTHESIS",
  "WIDTH=32",
  "MODE=fast sim",
]
```

宏名需要是合法标识符。带值宏会按 `NAME=value` 解析，`value` 可以包含空格。

### libraries

`libraries` 用于声明依赖库。路径相对当前项目配置文件所在目录解析。每个库路径都会被当作另一个工作区继续加载：

- 如果库目录下有 `vide.toml`，Vide 会按这个项目配置文件加载库。
- 如果库目录没有项目配置文件，Vide 会把该目录作为未配置库加载。
- 依赖会参与当前工程的项目分析，并支持传递依赖。

`libraries` 不会替代当前工程自己的 `sources`。当前工作区的 Verilog/SystemVerilog 源文件仍然应该通过 `sources` 声明；`libraries` 只描述被当前工程引用的外部工程或库目录。

### exclude

`exclude` 使用工作区相对 glob，适合排除生成目录、仿真输出目录、缓存目录或黑盒 stub：

```toml
exclude = ["build/**", "out/**", "sim/work/**", "**/*_bb.v"]
```

`exclude = ["build"]` 只匹配名为 `build` 的路径本身，不会递归排除目录内容；目录递归请写 `build/**`。

`exclude` 只影响 Vide 选择哪些文件进入项目分析。如果你使用 VS Code，并且还想让 VS Code 自己也少监听某些大目录，可以另外配置 VS Code 的 `files.watcherExclude`。这两个设置互不替代：前者决定 Vide 分析什么，后者决定 VS Code 文件监听器少关注什么。

## 项目文件发现规则

- 项目配置文件名必须是 `vide.toml`。
- Vide 不会从子目录自动发现项目配置文件。打开 `repo` 时，只检查 `repo/vide.toml`。
