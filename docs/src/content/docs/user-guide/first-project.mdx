Vide 打开任何 Verilog/SystemVerilog 项目后，都可以提供语法高亮和基础编辑体验。要让 Vide 更准确地做跨文件跳转、补全、诊断和宏分支判断，就需要在项目根目录放一个项目配置文件：`vide.toml`。

这页会先用一个小工程写出第一份 `vide.toml`，再教你给真实项目编写 `vide.toml`。完整字段和边界规则见 [项目配置参考](../project-configuration/)。

## 1. 创建项目配置文件

用 VS Code 打开 RTL 工程目录。如果目录里有 RTL 文件但没有 `vide.toml`，Vide 会提示创建项目配置文件。你也可以点击右下角 `Vide` 状态项，在状态菜单里选择 `创建项目配置文件`。

默认模板会创建一个空的项目配置：

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = []

# include_dirs = ["include"]
# defines = ["SYNTHESIS"]
# top_modules = ["top"]
# libraries = ["../common_cells"]
# exclude = ["build/**"]
```

:::note
- 默认模板里的 `sources = []` 只是一个待填写状态：它会阻止 Vide 直接扫描整个项目。把 `sources` 改成 `rtl/**` 这类真实路径后，Vide 才会按你的工程结构分析。
- 配置文件开头的 `#:schema ...` 是 TOML 注释。未来，它能够帮助编辑器识别配置格式。
:::

## 2. 按目录结构编写配置

假设你的工程长这样：

<FileTree>
- my-rtl/
  - vide.toml
  - rtl/
    - top.sv
    - counter.sv
  - include/
    - defs.svh
  - sim/
    - work/
  - build/
</FileTree>

对应的第一份配置可以写成：

```toml
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS"]
top_modules = ["top"]
exclude = ["build/**", "sim/work/**"]
```

这几项分别告诉 Vide：

- `sources`：**必填**，指示了哪些 RTL 源文件属于这个工程。例如 `rtl/**` 递归匹配目录下所有文件，或 `rtl/*.sv` 匹配该目录下一级所有的 `sv` 后缀的文件。
- `include_dirs`：**推荐填写**，提示 Vide `` `include "defs.svh" `` 这类头文件从哪里找。例如 `include`。
- `defines`：**可选**，预先打开哪些宏分支，例如可以写 `"SYNTHESIS"` 或 `"DATA_WIDTH=32"`。
- `top_modules`：**推荐填写**，当前工程的顶层模块名，提示 Vide 从哪里开始分析。
- `exclude`：**可选**，可以用于排除生成目录、仿真输出和缓存目录，Vide 不会分析这些目录。在本例中，`exclude` 字段也可以不写，因为 `sources` 不包含这些目录。

:::note
路径统一写 `/`，即使在 Windows 上也写 `rtl/**`，不要写 `rtl\**`。所有路径只能在工作区根目录中。
:::

## 3. 一个更接近真实工程的例子

上面的 `my-rtl` 是一个很小的工程，下面我们以一个真实工程 [darklife/darkriscv](https://github.com/darklife/darkriscv) 为例。仓库根目录有一份 `darkriscv.core`，里面已经描述了 fileset、target、顶层模块和工具参数。参考它，`vide.toml` 可以写成：

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json

top_modules = ["darksocv"]
sources = [
  "rtl/*.v",
]
include_dirs = [
  "rtl",
]
```

这里每个字段是这么来的：

- `top_modules = ["darksocv"]`：来自 `.core` 里 default target 的 `toplevel: darksocv`。
- `sources = ["rtl/*.v"]`：把 `rtl` 目录下的 Verilog 源文件加入分析。这个写法比 `.core` 里的 `rtl` fileset 稍宽一些，适合作为第一次配置；如果你想严格跟随 `.core`，也可以逐个列出 fileset 里的 Verilog 文件。
- `include_dirs = ["rtl"]`：`rtl/config.vh` 是通过反引号 `` `include `` 引入的头文件，所以把 `rtl` 加入 include 搜索目录。

`src/darksocv.mem` 是运行时通过 `$readmemh("../src/darksocv.mem", MEM, 0)` 读取的内存初始化文件，不是 `` `include `` 头文件。它不需要写进 `include_dirs`；如果仿真或综合脚本需要复制这个文件，仍由原来的工程脚本处理。

以上的配置文件只涵盖了可综合的部分，如果想连带仿真代码一起分析，我们应该怎么写 `vide.toml` 呢？

如果按 `.core` 里的 `sim` target 分析仿真入口，就在 default target 的基础上追加 `tb` fileset 里的 `sim/darksimv.v`，并把顶层从 `darksocv` 换成 `darksimv`。

这个 target 的默认仿真器是 Icarus Verilog。`sim/darksimv.v` 和 `rtl/config.vh` 会根据 `__ICARUS__` 选择仿真分支，所以这里把 `__ICARUS__` 写进 `defines`。`.core` 中的 `-g2012` 是传给 Icarus Verilog 的语言版本参数，不需要写进 `vide.toml`：

```toml
sources = [
  "rtl/*.v",
  "sim/darksimv.v",
]

include_dirs = [
  "rtl",
  "sim",
]

defines = ["__ICARUS__"]
top_modules = ["darksimv"]
```

板级 target 也按同样思路编写配置文件。例如 `colorlight_i5` target 使用 `rtl` 和 `colorlighti5` 两个 fileset，并通过 parameters 向综合工具传入 `__YOSYS__`、`LATTICE_ECP5_COLORLIGHTI5` 两个 Verilog 宏。对应配置可以写成：

```toml
sources = [
  "rtl/*.v",
  "boards/colorlighti5/pll_ref_25MHz.v",
]

include_dirs = ["rtl"]
defines = ["__YOSYS__", "LATTICE_ECP5_COLORLIGHTI5"]
top_modules = ["darksocv"]
```

## 4. 常见注意事项

- 项目配置文件名必须是 `vide.toml`。
- `vide.toml` 应放在 VS Code 打开的项目根目录。Vide 不会从子目录自动发现项目配置文件。

## 5. 项目状态

保存 `vide.toml` 后，Vide 默认会自动刷新项目配置。

实际使用时可以把项目状态分成三类：

- 没有 `vide.toml`：Vide 会尽量扫描项目，先让跳转、引用、悬停、符号信息等读代码功能可用，但不会创建项目级编译配置。
- 刚创建的空模板：默认写着 `sources = []`，表示暂时不扫描整个项目，等你填入真实源码路径。
- 写好 `vide.toml`：Vide 按 `sources`、`include_dirs`、`defines`、`libraries` 等字段理解工程。
