Vide 打开任何 Verilog/SystemVerilog 项目后，都可以提供语法高亮和基础的编辑体验。要让 Vide 更准确地做跨文件跳转、补全、诊断、宏分支判断和 Qihe 参数推导，就需要在工作区根目录放一个项目配置文件：`vide.toml`。

这页内容会指导大家进行项目的配置，完整字段和边界规则见 [项目配置参考](../project-configuration/)。

## 1. 创建项目配置文件

用 VS Code 打开 RTL 工程目录。如果目录里有 RTL 文件但没有 `vide.toml`，Vide 会提示创建项目配置文件。你也可以点击右下角 `Vide` 状态项，在状态菜单里选择 `创建项目配置文件`。

默认模板会创建一个空的项目配置：

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json
sources = []

# include_dirs = ["include"]
# defines = ["SYNTHESIS"]
# top_modules = ["top"]
# libraries = ["../common_cells"]
# exclude = ["build/**"]
```

:::note
- 默认模板里的 `sources = []` 只是一个待填写状态：它会阻止 Vide 直接扫描整个工作区。把 `sources` 改成 `rtl/**` 这类真实路径后，Vide 才会按你的工程结构分析。
- 配置文件开头的 `#:schema ...` 是 TOML 注释。未来，它能够帮助编辑器识别配置格式。
:::

## 2. 按目录结构编写配置

假设你的工程长这样：

<FileTree>
- my-rtl/
  - vide.toml
  - rtl/
    - top.sv
    - counter.sv
  - include/
    - defs.svh
  - sim/
    - work/
  - build/
</FileTree>

对应的第一份配置可以写成：

```toml
sources = ["rtl/**"]
include_dirs = ["include"]
defines = ["SYNTHESIS"]
top_modules = ["top"]
exclude = ["build/**", "sim/work/**"]
```

这几项分别告诉 Vide：

- `sources`：**必填**，指示了哪些 RTL 源文件属于这个工程。例如 `rtl/**` 递归匹配目录下所有文件，或 `rtl/*.sv` 匹配该目录下一级所有的 `sv` 后缀的文件。
- `include_dirs`：**推荐填写**，提示 Vide `` `include "defs.svh" `` 这类头文件从哪里找。例如 `include`。
- `defines`：**可选**，预先打开哪些宏分支，例如可以写 `"SYNTHESIS"` 或 `"DATA_WIDTH=32"`。
- `top_modules`：**推荐填写**，当前工程的顶层模块名，提示 Vide 从哪里开始分析。
- `exclude`：**可选**，可以用于排除生成目录、仿真输出和缓存目录，Vide 不会分析这些目录。在本例中，`exclude` 字段也可以不写，因为 `sources` 没有被 `exclude` 内容干扰。

:::note
路径统一写 `/`，即使在 Windows 上也写 `rtl/**`，不要写 `rtl\**`。所有路径只能在工作区根目录中。
:::

## 3. 一个更接近真实工程的例子

上面的 `my-rtl` 是一个很小的工程，下面我们以一个真实工程 [darklife/darkriscv](https://github.com/darklife/darkriscv) 为例。仓库根目录有一份 `darkriscv.core`，定义了项目信息，其对应的 `vide.toml` 可以写成：

```toml
#:schema https://vide.pascal-lab.net/schemas/v1/vide.schema.json

top_modules = ["darksocv"]
sources = [
  "rtl/*.v",
]
include_dirs = [
  "rtl",
  "src",
]
```

这里的对应关系是：

- `top_modules = ["darksocv"]`：来自 `.core` 里 default target 的 `toplevel: darksocv`。
- `sources = ["rtl/*.v"]`：覆盖 `darksocv` 直接或间接用到的 `darkpll.v`、`darkbridge.v`、`darkram.v`、`darkio.v`、`darkriscv.v`、`darkuart.v` 等模块定义。
- `include_dirs = ["rtl", "src"]`
  - `rtl/config.vh` 是 include 文件，因此 `rtl` 是 include 目录。
  - `src/darksocv.mem` 是仿真/初始化数据，在源码中被使用了，因此把 `src` 放进 `include_dirs`。

以上的配置文件只涵盖了可综合的部分，如果想连带仿真代码一起分析，我们应该怎么写 `vide.toml` 呢？

例如按 `.core` 里的 `sim` target 分析仿真入口，就把 `tb` fileset 里的 `sim/darksimv.v` 加进来，并把顶层换成 `darksimv`。`sim` target 的 icarus 配置里有 `-g2012`，而 `config.vh` 里会根据 `__ICARUS__` 切换仿真分支，所以这里也把宏写进 `defines`：

```toml
sources = [
  "rtl/*.v",
  "sim/darksimv.v",
]

include_dirs = [
  "rtl",
  "src",
  "sim",
]

defines = ["__ICARUS__"]
top_modules = ["darksimv"]
```

同理，板级 target 也按同样方式编写配置文件。例如 `colorlight_i5` target 使用 `rtl` 和 `colorlighti5` 两个 fileset，并启用 `__YOSYS__`、`LATTICE_ECP5_COLORLIGHTI5` 两个 vlog define。对应配置可以写成：

```toml
sources = [
  "rtl/*.v",
  "boards/colorlighti5/pll_ref_25MHz.v",
]

include_dirs = ["rtl", "src"]
defines = ["__YOSYS__", "LATTICE_ECP5_COLORLIGHTI5"]
top_modules = ["darksocv"]
```

## 4. 项目状态

保存 `vide.toml` 后，Vide 默认会自动刷新项目配置。

实际使用时可以把项目状态分成三类：

- 没有 `vide.toml`：Vide 会扫描工作区，先提供阅读类功能，但不会创建项目级编译配置。
- 刚创建的空模板：默认写着 `sources = []`，表示暂时不扫描整个工作区，等你填入真实源码路径。
- 写好 `vide.toml`：Vide 按 `sources`、`include_dirs`、`defines`、`libraries` 等字段理解工程。

具体来说，各行为是否配置项目的差异可以在各 [功能特性](../features/) 末尾看到。我们也总结了差异清单，请见 [项目配置前后的行为差异](../project-configuration/#项目配置前后的行为差异)。
