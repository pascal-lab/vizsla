## Qihe 与项目配置

- 修复 Windows 上 Qihe 启动时收到 `\\?\` verbatim cwd 路径的问题。Vizsla 现在会传递普通 Windows 路径，避免 Java 与外部工具链在解析工作目录时失败。（[#98](https://github.com/pascal-lab/vizsla/pull/98)）
- 从 VS Code 编辑器标题栏或文件菜单触发 Qihe 分析时，命令现在使用被点击文件的 URI，而不是误用当前 active editor；从命令面板或快捷键触发且没有资源参数时，仍回退到 active editor。（[#109](https://github.com/pascal-lab/vizsla/pull/109)）
- 新增 `vizsla.qihe.autoConfigureArgsFromManifest` 设置，默认保持原有行为。关闭后，Qihe 不再自动从 `vizsla.toml` 补充 `--mode sv`、`--top`、`-I`、`-D` 等 manifest-derived 参数；用户显式配置的 `vizsla.qihe.compileArgs` 与 `vizsla.qihe.runArgs` 仍会生效。（[#113](https://github.com/pascal-lab/vizsla/pull/113)）

## 诊断 Profiling

- 新增 `Vizsla: Profile Diagnostics` 命令，可对当前文件或 workspace diagnostics 启动独立 profiling language-server session，并生成 trace summary、folded stacks、静态 flamegraph 与原始 Chrome trace JSON。（[#110](https://github.com/pascal-lab/vizsla/pull/110)）
- `Vizsla: Profile Diagnostics` 入口迁移到统一的 Vizsla 状态菜单，不再占用编辑器标题栏；用户仍可从状态菜单启动当前文件或 workspace diagnostics profiling。（[#118](https://github.com/pascal-lab/vizsla/pull/118)）
- 交互式 profiling 视图改为使用打包在 VSIX 内的 Speedscope 资源，并在外部浏览器打开，避免大型 trace 渲染占用 VS Code webview 进程资源。（[#110](https://github.com/pascal-lab/vizsla/pull/110)）
- profiling session 现在只继承启动专用 language server 所需的配置，避免把临时 server 启动参数泄漏到普通扩展会话。（[#114](https://github.com/pascal-lab/vizsla/pull/114)）

## LSP 性能与正确性

- vendored slang 源码并修复 Windows/MSVC Release 构建优化参数，使 slang C++ 代码按优化构建；在测试场景中，默认诊断耗时从约 82s 降到约 16.6s。（[#102](https://github.com/pascal-lab/vizsla/pull/102), [#103](https://github.com/pascal-lab/vizsla/pull/103)）
- 诊断路径现在以 profile-level slang compilation 作为 parse diagnostics 与 semantic diagnostics 的共同来源，避免对同一 compilation root 重复 parse/compile，并启用 slang 默认 warning group。在同一测试场景中，默认诊断耗时进一步降到约 9.8s；显式 `warnings=["everything"]` 仍会报告完整 warning 集合。（[#104](https://github.com/pascal-lab/vizsla/pull/104)）
- preprocessor index 构建避免触发完整 parse，降低诊断与索引过程中的重复工作。（[#105](https://github.com/pascal-lab/vizsla/pull/105)）
- 修复 named port 与 parameter connections 的解析作用域。跳转定义与语义 token 现在会解析到被实例化目标 module 的端口或参数，而不是当前 module scope。（[#107](https://github.com/pascal-lab/vizsla/pull/107)）
- 修复系统任务补全 snippet 中 `$` 被 VS Code 当作 snippet 变量处理的问题。接受 `$display` 等补全项时会保留任务名。（[#106](https://github.com/pascal-lab/vizsla/pull/106)）

## VS Code 体验

- 改进 Verilog/SystemVerilog TextMate grammar 覆盖。系统任务高亮现在基于 slang 已知系统任务列表，`.v`/`.vh` 保持 Verilog grammar，`.sv`/`.svh`/`.svi` 使用 SystemVerilog grammar。（[#108](https://github.com/pascal-lab/vizsla/pull/108)）
- 语言服务器或 workspace reload 等 VS Code 侧错误提示现在提供“显示输出”操作，Qihe 分析失败会提供“显示 Qihe 输出”操作，并让临时 Qihe 状态入口指向对应日志面板，方便直接查看失败原因。（[#117](https://github.com/pascal-lab/vizsla/pull/117)）

## 内部与维护

- 将 VS Code 扩展开发依赖中的 `qs` 更新到 `6.15.2`，包含上游针对 `parse` 与 `stringify` 的回归修复。（[#112](https://github.com/pascal-lab/vizsla/pull/112)）
- 修复 `master` 上由 Clippy `obfuscated_if_else` 触发的 Rust lint 失败。（[#115](https://github.com/pascal-lab/vizsla/pull/115)）
- 发布 VS Code 扩展 `0.1.4`。
