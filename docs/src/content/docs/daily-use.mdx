这一章按日常编辑 RTL 的顺序介绍 Vizsla。它不是完整设置参考；需要逐项配置时请看 [VS Code 设置](./vscode-settings.md)，需要告诉 Vizsla 项目文件在哪里时请看 [项目配置](./project-configuration.md)。

## 语言识别

VS Code 扩展会注册两类语言：

| 语言 | 文件扩展名 |
| --- | --- |
| Verilog | `.v`, `.vh` |
| SystemVerilog | `.sv`, `.svh`, `.svi` |

打开这些文件后，VS Code 会先应用 Vizsla 的基础语法高亮和语言配置。语法高亮、注释规则和括号匹配不依赖后台语言服务器启动成功。

后台语言服务器启动后，诊断（`Problems` 面板里的错误、警告和提示）、跳转、补全、悬停说明（Hover）、签名帮助（signature help）、行内提示（inlay hints）、语义高亮（semantic tokens）、大纲、折叠、扩大/缩小选择和实例数量提示（code lens）才会接入。跨文件能力需要当前工作区（你用 VS Code 打开的顶层目录）有足够的项目信息。

## 诊断

日常编辑时，诊断会显示在 VS Code 的 `Problems` 面板和编辑器下划线里。Vizsla 会提供两类诊断：单文件语法问题，以及需要项目配置才能判断的跨文件问题。

默认诊断是启用的，刷新时机是 `vizsla.diagnostics.update = "onSave"`，也就是保存后刷新。大型 RTL 工程通常适合保留这个默认值；如果确实需要边输入边刷新，可以在设置里改成 `"onType"`。

快速修复（quick fix）通常跟随相关诊断出现。看不到预期修复时，先确认对应诊断已经刷新出来，再检查目标模块、include 目录、宏定义和库文件是否已经被当前项目配置加载。

如果某一类来自 slang 的诊断在你的项目里需要隐藏或降级，灯泡里也可能出现“忽略此类诊断”或“降级为 warning”的快速修复；这些操作会写入用户设置或工作区设置。完整设置见 [VS Code 设置的 Diagnostics](./vscode-settings.md#diagnostics)。项目文件如何影响跨文件诊断，请看 [项目配置](./project-configuration.md) 和 [解析与分析模型](./parsing-and-analysis.md)。

## 导航和阅读

阅读 RTL 时，常用入口是 `Go to Definition`、`Go to Declaration`、`Find References`、当前文件高亮和重命名：

- 在实例名上跳到目标模块。
- 在信号、端口、参数、typedef、函数或任务上跳到声明/定义。
- 查找同一符号在当前工程里的引用。
- 光标停在符号上时，高亮当前文件里的相关位置。
- 需要改名时，VS Code 会先判断当前位置是否可重命名，再基于引用搜索修改相关位置。

这些能力比纯文本搜索更依赖项目信息。结果不符合预期时，通常先检查目标文件是否被项目配置加载，以及 `vizsla.scope.visibility` 是否符合你的作用域习惯。

## 补全、提示和快速修复

补全会在常见 Verilog/SystemVerilog 输入位置出现，例如模块正文、`.` 后的成员、实例端口列表、参数列表、事件控制、预处理指令和系统任务/函数附近。

Vizsla 会按当前位置选择补全来源，而不是把所有候选都塞进列表。常见结果包括预处理指令、关键字和代码片段（snippets）、表达式候选、`.` 后的成员、实例端口/参数、事件控制里的信号和系统任务/函数。注释、字符串和字面量里会尽量避免无关候选。端口/参数补全需要 Vizsla 能找到被实例化的目标模块。

Hover 会优先识别名字和字面量：

- 对符号名，我们会解析定义并渲染模块、端口、参数、声明、实例、函数等信息。
- 对端口连接 shorthand，我们会同时展示 port 和 local 两侧的信息。
- 对字面量，我们会渲染解析后的 literal 信息。

签名帮助（signature help）主要用于模块实例端口连接和参数赋值列表：你输入连接时可以看到目标模块的端口/参数顺序，当前参数会随光标位置更新。只想显示参数相关签名时，使用 `vizsla.signature.help.params.only`。

行内提示（inlay hints）默认启用，用来减少反复查端口表的机械动作：

- 有序端口连接或空连接显示目标端口名。
- 有序参数赋值显示目标参数名。
- 结构结尾显示结构名。

快速修复和重构操作会跟它能解决的问题一起出现：

- 如果 `Problems` 面板提示实例端口没接全，例如未连接端口或未命名端口，光标放到实例连接列表附近时，灯泡里会出现 `补全连接`。
- 如果参数没有值，灯泡里会出现 `补全参数`。
- 如果端口或参数混用了有序写法和命名写法，可以用 `将有序端口连接转换为命名连接`、`将有序参数赋值转换为命名赋值`，或在端口列表里用 `移除空端口连接` 清理空连接。
- 如果写了 `.port` 但缺少 `()`，可以用 `添加显式空端口连接`。如果实例完全缺少端口列表，可以用 `添加空实例端口列表`。
- 光标在整数常量上时，还可能出现 `将字面量转换为二进制`、`将字面量转换为十进制`、`将字面量转换为十六进制` 等进制转换。

这些操作需要 Vizsla 能理解当前位置。实例端口/参数相关操作还需要目标模块能被解析到；如果目标模块本身找不到，请先修正 `sources`、`include_dirs`、`defines` 或 `libraries`。当前没有专门“自动修复找不到模块或包导入”的快速修复，这类问题要先把项目配置补完整。

格式化入口包括 `Format Document`、`Format Selection` 和按 Enter 时的自动缩进行为。当前格式化工具只有 `verible`，会调用外部 `verible-verilog-format`；未在 `PATH` 中时，在 `vizsla.formatter.path` 配置绝对路径。格式化参数和 Enter 行为设置见 [VS Code 设置的 Formatter 和 Formatting](./vscode-settings.md#formatter-和-formatting)。

## 视觉和结构辅助

语义高亮（semantic tokens）会在基础语法高亮之外补充“这是端口、这是时钟/复位、这是读写位置”等信息。默认设置会增强时钟/复位端口、输入/输出端口和读写位置的显示。主题支持时，端口方向、时钟复位和普通符号会更容易区分。

大纲（Outline）和折叠面向长文件阅读。大纲会展示模块、端口、参数、声明、实例、block、function、generate 等符号；折叠会收起模块、块、语句和注释等结构。

扩大/缩小选择支持 VS Code 的相关命令，可以从当前 token 逐步扩展到表达式、语句、块或更大的语法结构。

实例数量提示（code lens）默认启用，在模块声明处显示 `0 instances`、`1 instance` 或 `N instances`。当前提示只显示实例数量，不绑定跳转命令；需要定位具体实例时使用 `Find References`。

## 常见前提和限制

完整跨文件体验依赖项目配置。VS Code 创建的默认 `vizsla.toml` 会写入 `sources = []`，不会扫描整个工作区；想启用更准确的跨文件诊断、端口/参数补全和实例相关操作，请按工程结构补 `sources`、`include_dirs`、`defines`、`libraries` 或 `top_modules`。细节见 [项目配置](./project-configuration.md)。

缺少 include 目录、宏定义、库或目标模块时，Vizsla 会尽量保留单文件能力，但跨文件诊断、实例签名、端口/参数补全、行内提示、快速修复和实例数量提示都可能减少或消失。

日常排障可以从命令面板里的 `Vizsla：显示状态`、`Vizsla：显示语言服务器输出`、`Vizsla：重新加载项目配置`、`Vizsla：重启语言服务器` 和 `Vizsla：显示服务器版本` 开始。Qihe 分析和诊断性能分析命令见 [命令、状态和日志](./commands-status-logs.md)。

如果功能和这里描述不一致，优先看 [故障排查](./troubleshooting.md)：它覆盖服务器未启动、工程未加载、诊断未刷新、格式化失败和 watcher 刷新问题。
