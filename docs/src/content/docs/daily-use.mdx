这一章按功能特性介绍 Vizsla：先确认文件被识别，再看诊断、跳转、补全、快速修复和格式化。需要逐项配置时请看 [VS Code 设置](./vscode-settings.md)；需要告诉 Vizsla 项目文件在哪里时请看 [项目配置](./project-configuration.md)。

## 语言识别

VS Code 扩展会注册两类语言：

| 语言 | 文件扩展名 |
| --- | --- |
| Verilog | `.v`, `.vh` |
| SystemVerilog | `.sv`, `.svh`, `.svi` |

打开这些文件后，VS Code 会先应用 Vizsla 的基础语法高亮和语言配置。语法高亮、注释规则和括号匹配不依赖后台语言服务器；即使服务器还没启动成功，这些基础编辑体验也应该可用。

语言服务器启动后，`Problems` 面板里的诊断、跳转、补全、悬停说明、签名帮助、行内提示、语义高亮、大纲、折叠、扩大/缩小选择和实例数量提示才会接入。需要跨文件理解时，当前工作区，也就是你用 VS Code 打开的顶层目录，需要有足够的项目配置。

## 诊断

日常编辑时，诊断会显示在 VS Code 的 `Problems` 面板和编辑器下划线里。Vizsla 会提供两类诊断：当前文件就能判断的语法问题，以及需要项目配置才能判断的跨文件问题。

默认诊断是启用的，刷新时机是 `vizsla.diagnostics.update = "onSave"`，也就是保存后刷新。大型 RTL 工程通常适合保留这个默认值；如果确实需要边输入边刷新，可以在设置里改成 `"onType"`。

快速修复通常跟随相关诊断出现。看不到预期修复时，先确认对应诊断已经刷新出来，再检查目标模块、include 目录、宏定义和库文件是否已经被当前项目配置加载。

如果某一类来自 slang 的诊断在你的项目里需要隐藏或降级，灯泡里也可能出现“忽略此类诊断”或“降级为警告”的快速修复；这些操作会写入用户设置或工作区设置。完整设置见 [VS Code 诊断设置](./vscode-settings.md#diagnostics)。项目文件如何影响跨文件诊断，请看 [项目配置](./project-configuration.md) 和 [无配置项目的分析方式](./parsing-and-analysis.md)。

## 导航和阅读

阅读 RTL 时，最常用的是跳转、查引用、当前文件高亮和重命名：

- 在实例名上执行 `Go to Definition`，跳到目标模块。
- 在信号、端口、参数、typedef、函数或任务上执行 `Go to Declaration` 或 `Go to Definition`，跳到声明或定义。
- 执行 `Find References`，查找同一符号在当前工程里的引用。
- 光标停在符号上时，当前文件里的相关位置会一起高亮。
- 需要改名时，执行重命名；VS Code 会先判断当前位置是否可重命名，再基于引用搜索修改相关位置。

这些能力比纯文本搜索更依赖项目信息。结果不符合预期时，通常先检查目标文件是否被项目配置加载，以及 `vizsla.scope.visibility` 是否符合你的作用域习惯。

## 补全、提示和快速修复

补全会在常见 Verilog/SystemVerilog 输入位置出现，例如模块正文、`.` 后的成员、实例端口列表、参数列表、事件控制、预处理指令和系统任务/函数附近。

Vizsla 会按当前位置选择补全来源，而不是把所有候选都塞进列表。常见结果包括预处理指令、关键字、代码片段、表达式候选、`.` 后的成员、实例端口或参数、事件控制里的信号和系统任务/函数。注释、字符串和字面量里会尽量避免无关候选。端口和参数补全需要 Vizsla 能找到被实例化的目标模块。

悬停说明会优先识别名字和字面量：

- 对符号名，Vizsla 会解析定义并展示模块、端口、参数、声明、实例、函数等信息。
- 对端口连接简写，Vizsla 会同时展示端口和本地信号两侧的信息。
- 对字面量，Vizsla 会展示解析后的信息。

签名帮助主要用于模块实例端口连接和参数赋值列表：你输入连接时可以看到目标模块的端口或参数顺序，当前项会随光标位置更新。只想显示参数相关签名时，使用 `vizsla.signature.help.params.only`。

行内提示默认启用，用来减少反复查端口表的机械动作：

- 有序端口连接或空连接显示目标端口名。
- 有序参数赋值显示目标参数名。
- 结构结尾显示结构名。

快速修复和重构操作会跟它能解决的问题一起出现。常见情况有：

- 如果 `Problems` 面板提示实例端口没接全，例如未连接端口或未命名端口，光标放到实例连接列表附近时，灯泡里会出现 `补全连接`。
- 如果参数没有值，灯泡里会出现 `补全参数`。
- 如果端口或参数混用了有序写法和命名写法，可以用 `将有序端口连接转换为命名连接`、`将有序参数赋值转换为命名赋值`，或在端口列表里用 `移除空端口连接` 清理空连接。
- 如果写了 `.port` 但缺少 `()`，可以用 `添加显式空端口连接`。如果实例完全缺少端口列表，可以用 `添加空实例端口列表`。
- 光标在整数常量上时，还可能出现 `将字面量转换为二进制`、`将字面量转换为十进制`、`将字面量转换为十六进制` 等进制转换。

这些操作需要 Vizsla 能理解当前位置。实例端口或参数相关操作还需要目标模块能被解析到；如果目标模块本身找不到，请先修正 `sources`、`include_dirs`、`defines` 或 `libraries`。当前没有专门“自动修复找不到模块或包导入”的快速修复，这类问题要先把项目配置补完整。

## 视觉和结构辅助

语义高亮会在基础语法高亮之外补充“这是端口、这是时钟/复位、这是读写位置”等信息。默认设置会增强时钟/复位端口、输入/输出端口和读写位置的显示。主题支持时，端口方向、时钟复位和普通符号会更容易区分。

大纲和折叠面向长文件阅读。大纲会展示模块、端口、参数、声明、实例、block、function、generate 等符号；折叠会收起模块、块、语句和注释等结构。

扩大/缩小选择支持 VS Code 的相关命令，可以从当前 token 逐步扩展到表达式、语句、块或更大的语法结构。

实例数量提示默认启用，在模块声明处显示 `0 instances`、`1 instance` 或 `N instances`。当前提示只显示实例数量，不绑定跳转命令；需要定位具体实例时使用 `Find References`。

## 常见前提和限制

完整跨文件体验依赖项目配置。VS Code 创建的默认 `vizsla.toml` 会写入 `sources = []`，不会扫描整个工作区；想启用更准确的跨文件诊断、端口/参数补全和实例相关操作，请按工程结构补 `sources`、`include_dirs`、`defines`、`libraries` 或 `top_modules`。细节见 [项目配置](./project-configuration.md)。

缺少 include 目录、宏定义、库或目标模块时，Vizsla 会尽量保留单文件能力，但跨文件诊断、实例签名、端口/参数补全、行内提示、快速修复和实例数量提示都可能减少或消失。

日常排障可以从命令面板里的 `Vizsla：显示状态`、`Vizsla：显示语言服务器输出`、`Vizsla：重新加载项目配置`、`Vizsla：重启语言服务器` 和 `Vizsla：显示服务器版本` 开始。Qihe 分析和诊断性能分析命令见 [命令、状态和日志](./commands-status-logs.md)。

如果功能和这里描述不一致，优先看 [故障排查](./troubleshooting.md)：它覆盖服务器未启动、工程未加载、诊断未刷新、格式化失败和文件监听刷新问题。
